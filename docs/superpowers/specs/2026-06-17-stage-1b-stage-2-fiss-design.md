# Stage 1B + Stage 2 FISS Compression Tools Design Spec

## Source And Status

This spec is derived from `stage_2_design.txt` and assumes the current Stage 1A implementation is present in `src/simulation/elements/`.

Stage 1A status:
- `FISS` stores fictional compression in `tmp`.
- `FISS` stores fictional neutron activation in `tmp2`.
- Compressed `FISS` can emit `NTRN`.
- Compressed and activated `FISS` can fission into `FPRD`, producing heat, pressure, neutrons, optional `SHOK`, and local chain-kicks.
- `CMTR`, `HEXP`, `SHOK`, and `DENS` exist as first-pass support elements.

Stage 1B + Stage 2 must preserve the successful core `FISS` behavior. Do not redesign `FISS` unless needed for compatibility, tuning, or stability.

## Safety And Scope

This is a fictional particle sandbox mechanic only. All values must be arbitrary gameplay constants.

Do not add or model:
- real isotope names or real nuclear data,
- real explosive data or explosive lensing,
- real device geometry or compression ratios,
- critical mass, tampers, reflectors, staged devices, or yield calculations,
- fusion fuels, lithium materials, deuterium/tritium materials,
- x-rays, gamma rays, moderators, or neutron absorber physics,
- real material densities, cross-sections, or engineering formulas.

The intended gameplay distinction is:
- `HEXP`, `BEXP`, `SEXP`, `SHOK`, and `PDRV` are compression drivers.
- `PABS`, `PWAL`, and `DENS` are inert pressure-control test materials.
- `FISS` is the compact high-energy reaction source.

## Design Goals

1. Hotfix `HEXP` and `SHOK` so pressure tools create useful compression without becoming the main dramatic event.
2. Improve diagnostics and tuning for `FISS`, `NTRN`, `FPRD`, and `CMTR`.
3. Add fictional pressure/compression tools that let players shape pressure in gamey, non-realistic ways.
4. Preserve strict caps and safety behavior so reactions remain tunable and do not create unlimited particles, heat, or pressure.

## Recommended Approach

Use an incremental implementation sequence:

1. Rebalance `HEXP` and `SHOK`.
2. Compile and manually smoke-test pressure and heat output.
3. Improve `CMTR`, tuning organization, counters, and stability safeguards.
4. Compile and manually test diagnostics.
5. Add `BEXP` and `SEXP`.
6. Compile and manually compare their pressure profiles.
7. Add `PDRV`, `PABS`, and `PWAL` if still manageable.
8. Compile and run the full manual Stage 1B + Stage 2 test pass.

This keeps the highest-risk behavioral correction first: `HEXP` must stop overwhelming the `FISS` reaction system.

## Tuning Organization

Keep all fictional compression-system constants in one obvious tuning area. The current `src/simulation/elements/FISS.h` may continue to serve this role, or it may be renamed later if the implementation wants a broader name such as `FissCompressionTuning.h`.

The tuning section must group constants by behavior:
- `FISS` compression gain, decay, thresholds, emission, fission, heat, pressure, and chain-kick.
- `NTRN` lifetime, velocity, activation, and absorption.
- `FPRD` heat, cooling, weak neutron emission, and decay.
- `HEXP`, `BEXP`, `SEXP`, `SHOK`, and `PDRV` pressure/heat/life values.
- `PABS`, `PWAL`, and `DENS` resistance/damping values.
- Global per-frame caps.

All new constants must be named plainly and be easy to tune without hunting through element update code.

## Part A: HEXP And SHOK Hotfix

### HEXP

`HEXP` is a fictional high-pressure test explosive. It exists to compress `FISS`; it must not be the primary high-energy event.

Required behavior:
- Produces a useful but capped pressure pulse.
- Produces modest heat only.
- Does not directly create plasma.
- Does not create extreme temperatures by itself.
- Converts to a mild byproduct such as `SMKE`, `GAS`, or empty space after detonation.
- Can push nearby `FISS.tmp` upward through pressure and nearby `SHOK`.
- Does not automatically cause a large reaction unless compressed `FISS` takes over.

Required constants:
- `HEXP_PRESSURE`
- `HEXP_PRESSURE_CAP`
- `HEXP_HEAT`
- `HEXP_MAX_TEMP`
- `HEXP_RADIUS`
- `HEXP_SHOK_COUNT`

Implementation notes:
- Keep `HEXP` ignition compatible with normal TPT conventions: heat, spark, nearby flame, nearby plasma, lightning, or `SHOK`.
- Cap pressure additions locally, not just globally.
- Clamp generated byproduct temperature to `HEXP_MAX_TEMP`.
- Add a code comment stating that `HEXP` is a fictional compression driver, not the high-energy reaction material.

### SHOK

`SHOK` is a short-lived fictional shock/compression effect. It should be the visible helper for compression testing.

Required behavior:
- Adds pressure with little or no heat.
- Disappears quickly.
- May directly nudge adjacent `FISS.tmp` slightly, but pressure-field behavior should remain the main mechanism.
- Does not recursively create more `SHOK` except through already capped callers.
- Remains visually distinct without resembling the main high-energy event.

Required constants:
- `SHOK_PRESSURE`
- `SHOK_PRESSURE_CAP`
- `SHOK_HEAT`
- `SHOK_LIFE`
- `SHOK_RADIUS`
- `SHOK_DECAY`

Implementation notes:
- Use `SHOK_HEAT` as zero or very small by default.
- Cap local pressure additions with `SHOK_PRESSURE_CAP`.
- Keep `PROP_LIFE_DEC | PROP_LIFE_KILL_DEC`.

## Part B: Stage 1B Diagnostics, Tuning, And Stability

### CMTR

`CMTR` should make tuning compressed `FISS` setups easier.

Required baseline display:
- Shows average nearby `FISS.tmp` compression.
- Shows average nearby `FISS.tmp2` activation.
- Indicates whether nearby `FISS` is above the neutron-emission threshold.

Preferred mode support:
- Mode 0: average nearby `FISS` compression.
- Mode 1: average nearby `FISS` activation.
- Mode 2: nearby `NTRN` flux.
- Mode 3: nearby fission activity.
- Mode 4: local pressure.

Mode selection may use `tmp`, `ctype`, or another existing property convention. If mode switching is too much for this pass, implement the improved baseline display and leave modes for a later pass.

Visible output may use color, `temp`, `life`, `tmp`, `tmp2`, or another existing game convention. The most important requirement is that compressed and neutron-ready `FISS` is visually obvious.

### Counters And Debug Values

Expose or internally maintain these counters if practical:
- fissions this frame,
- `NTRN` created this frame,
- `SHOK` created this frame,
- `FISS` neutron emissions this frame,
- compressed `FISS` count,
- active `FISS` count.

The current frame-cap state may be extended so `CMTR` mode 3 can detect recent local fission activity. Avoid adding expensive full-map scans unless necessary.

### Stability Safeguards

Keep or enforce:
- `MAX_FISSION_EVENTS_PER_FRAME`
- `MAX_NTRN_CREATED_PER_FRAME`
- `MAX_SHOK_CREATED_PER_FRAME`
- `MAX_FISS_NEUTRON_EMISSIONS_PER_FRAME`
- local pressure caps,
- local heat caps,
- bounds checks for particle and pressure-cell access,
- safe particle creation checks,
- no recursive unlimited particle spawning.

One small event must not be able to create unlimited particles, unbounded heat, or unbounded pressure.

### FPRD

`FPRD` should feel hot and dangerous but should not become the main runaway source.

Required behavior:
- Cools over time.
- May emit weak `NTRN` at low probability.
- Eventually becomes inert ash/debris if this remains easy and stable.

If current `FPRD` behavior is already stable, keep changes minimal.

## Part C: Stage 2 Fictional Pressure / Compression Tools

### BEXP

`BEXP` is a brisant-style fictional pressure explosive.

Required behavior:
- Produces a sharper, shorter pressure spike than `HEXP`.
- Produces little or modest heat.
- Uses capped pressure.
- May create a small capped number of `SHOK` particles.
- Converts to mild byproduct or empty space after activation.
- Useful for rapid `FISS` compression.

Required constants:
- `BEXP_PRESSURE`
- `BEXP_PRESSURE_CAP`
- `BEXP_HEAT`
- `BEXP_RADIUS`
- `BEXP_SHOK_COUNT`

### SEXP

`SEXP` is a slow-push fictional pressure explosive.

Required behavior:
- Lower peak pressure than `BEXP`.
- Longer pressure duration.
- Better for sustained compression.
- Produces modest heat only.
- Uses `life` or another simple timer for duration.
- May create a small capped number of `SHOK` particles.
- Converts to mild byproduct or empty space when finished.

Required constants:
- `SEXP_PRESSURE`
- `SEXP_PRESSURE_CAP`
- `SEXP_HEAT`
- `SEXP_RADIUS`
- `SEXP_DURATION`
- `SEXP_SHOK_COUNT`

### PDRV

`PDRV` is an abstract directional pressure driver.

Required behavior:
- Fictional test element that biases pressure in a preferred direction.
- May direct `SHOK` particles or add directional pressure bias based on `tmp`, `ctype`, or local velocity.
- Must not encode real shaped-charge, explosive lens, or weapon-geometry behavior.
- Has limited lifetime or activation behavior.
- Uses capped pressure and capped `SHOK` creation.

Required constants:
- `PDRV_PRESSURE`
- `PDRV_PRESSURE_CAP`
- `PDRV_DIRECTION_MODE`
- `PDRV_SHOK_COUNT`
- `PDRV_LIFE`

Suggested direction model:
- `tmp` stores direction mode or angle bucket.
- Direction mode maps to simple cardinal or diagonal pressure bias.
- The mapping is arbitrary and game-oriented.

### PABS

`PABS` is an inert pressure absorber/damper.

Required behavior:
- Reduces local pressure spikes.
- Does not emit radiation or neutrons.
- Does not trigger `FISS`.
- Does not become a neutron absorber/moderator mechanic.
- Helps players tune pressure behavior and prevent runaway pressure.

Required constants:
- `PABS_PRESSURE_DAMPING`
- `PABS_HEAT_RESISTANCE`

### PWAL

`PWAL` is an inert pressure-resistant wall/material.

Required behavior:
- Solid and generic.
- More pressure/heat resistant than ordinary materials.
- Useful for testing pressure containment.
- Not a tamper, reflector, or real component.
- Does not emit or reflect neutrons beyond generic engine behavior.

Required constants:
- `PWAL_PRESSURE_RESISTANCE`
- `PWAL_HEAT_RESISTANCE`

### DENS

Keep `DENS` as dense inert material for testing pressure/confinement.

Required behavior:
- Does not emit neutrons.
- Does not reflect neutrons as a special mechanic.
- Does not trigger fission.
- Does not become part of real-world device modelling.

## Manual Tests

### Test 1: HEXP Hotfix

Setup: detonate `HEXP` alone.

Expected:
- visible pressure pulse,
- modest heat,
- no plasma byproduct from `HEXP` itself,
- no extreme temperature event.

### Test 2: HEXP Near Ordinary Materials

Setup: detonate `HEXP` near common solids, powders, and gases.

Expected:
- blast/compression behavior,
- no excessive heat destruction,
- no plasma fireball.

### Test 3: HEXP Near FISS

Setup: detonate `HEXP` near a small `FISS` blob.

Expected:
- nearby `FISS.tmp` rises,
- dramatic reaction only happens if the `FISS` compression and activation system takes over.

### Test 4: SHOK Compression

Setup: place or create `SHOK` near `FISS`.

Expected:
- `FISS.tmp` increases,
- little or no heat is added,
- `SHOK` disappears quickly.

### Test 5: BEXP vs HEXP

Setup: compare `BEXP` and `HEXP` near equivalent `FISS` blobs.

Expected:
- `BEXP` gives sharper, shorter compression,
- `HEXP` gives broader general compression,
- neither produces the main high-energy event by itself.

### Test 6: SEXP vs BEXP

Setup: compare `SEXP` and `BEXP` near equivalent `FISS` blobs.

Expected:
- `SEXP` gives lower but longer compression,
- `BEXP` gives sharper compression.

### Test 7: FISS Reaction Source

Setup: compress `FISS` using `HEXP`, `BEXP`, or `SEXP`.

Expected:
- major heat, pressure, neutron output, and fission products come from `FISS`, not the pressure tool.

### Test 8: CMTR Diagnostics

Setup: place `CMTR` near `FISS` during compression.

Expected:
- `CMTR` clearly shows compression,
- `CMTR` clearly shows activation,
- `CMTR` clearly indicates neutron-emission readiness.

### Test 9: Pressure Control

Setup: use `PABS`, `PWAL`, and `DENS` near pressure tools.

Expected:
- pressure behavior is easier to control and observe,
- control materials remain inert and generic.

### Test 10: Stability

Setup: build a moderate `FISS` plus pressure-tool setup.

Expected:
- no crash,
- no freeze,
- no unlimited particle spawning,
- no unbounded pressure or heat.

## Implementation Reporting Requirements

After implementation, report:
- files changed,
- elements modified,
- elements added,
- constants added or changed,
- how `HEXP` was rebalanced,
- how to tune pressure without increasing heat,
- how to test Stage 1B + Stage 2,
- known limitations and follow-up work.

## Acceptance Criteria

Stage 1B + Stage 2 is acceptable when:
- `HEXP` no longer creates extreme heat, plasma, or overwhelming pressure by itself.
- `SHOK` is a short-lived low-heat pressure helper.
- `FISS` remains the main source of dramatic heat, pressure, neutron, and fission-product output.
- `CMTR` gives clearer compression and activation feedback.
- `BEXP` and `SEXP` provide distinct pressure profiles.
- `PDRV`, `PABS`, and `PWAL` are fictional pressure tools/materials with no real-world weapon-design logic.
- The build compiles.
- The manual tests above have been run or explicitly deferred with reasons.
